The Other Half of Visibility: Endpoint Detection with Wazuh

2026-07-1818 min

In the last post we built network visibility — Zeek watching the wire, feeding Elasticsearch and Kibana. That tells you what crossed the network. What it can't tell you is what happened on a machine: a config file quietly edited, a new admin account created at 3 a.m., a program that started acting strange, a missing security patch. For that, you need eyes on the endpoint itself.

That's what Wazuh does. It's a free, open-source security platform that puts a lightweight agent on each machine you care about — laptops, servers, VMs — and reports back on:

  • File integrity — tells you when important files change, and what changed.
  • Log analysis — reads the endpoint's own logs and raises alerts on suspicious events (failed logins, privilege escalation, and so on).
  • Security configuration assessment — scores each machine against hardening benchmarks and tells you exactly what's weak.
  • Vulnerability detection — cross-references installed software against known CVEs.
  • Active response — can react automatically to certain events (block an IP, kill a process) if you choose to enable it.

Everything below is a full walkthrough. As with the network guide, it uses example addresses like 10.0.0.5 and generic hostnames — swap in your own. Nothing here is specific to any one environment.

How this relates to the last post

A fair question: didn't we just install Elasticsearch and Kibana? Does Wazuh reuse them?

Not directly. Wazuh ships its own bundled storage and dashboard (built on the OpenSearch lineage, a cousin of Elasticsearch), so it's completely self-contained — you don't need the stack from the last post for it to work. You can wire Wazuh's alerts into an existing Elastic setup if you want everything in one pane later, but that's an advanced move. For a clean start, we'll use Wazuh's all-in-one install, which stands up the whole thing with a single command.

Think of it as two complementary sensors: Zeek watches the network, Wazuh watches the machines. Together they cover the two places security problems show up.

What you'll need

  • A server for the Wazuh central components. Like Elasticsearch, the indexer under Wazuh is memory-hungry — give it at least 8 GB of RAM, 2+ CPU cores, and some disk. A separate box or VM from your network sensor is cleanest.
  • A 64-bit Linux host for that server — Ubuntu or a compatible distribution, freshly installed.
  • One or more endpoints to monitor — any Linux, Windows, or macOS machine you want eyes on. We'll cover Linux and Windows agents below.
  • sudo/administrator access on everything, and about half an hour.

Part 1 — Install the Wazuh central stack

Wazuh provides an assisted installer that puts all three central components — the server (manager), the indexer, and the dashboard — on one host. On your server box:

curl -sO https://packages.wazuh.com/4.14/wazuh-install.sh
sudo bash ./wazuh-install.sh -a

The -a flag means "all-in-one." The installer downloads the components, generates TLS certificates, sets everything up, and takes roughly 5–15 minutes depending on your connection and hardware.

When it finishes, it prints your credentials — capture them. The tail of the output looks like this:

INFO: --- Summary ---
INFO: You can access the web interface https://<YOUR_SERVER_IP>
    User: admin
    Password: <A_GENERATED_PASSWORD>

Save that password. It's also written to an archive the installer leaves behind, in case you lose it:

sudo tar -O -xvf wazuh-install-files.tar wazuh-install-files/wazuh-passwords.txt

Part 2 — Log into the dashboard

Open a browser and go to https://<YOUR_SERVER_IP>. The dashboard runs over HTTPS on the standard port with a self-signed certificate, so your browser will warn you the first time — that's expected on a fresh install; accept it and continue.

Log in as admin with the generated password. You'll land on the Wazuh overview. It's empty right now because no machines are reporting yet. Let's fix that.

Security note: that self-signed certificate and the admin account are fine for a lab on a trusted network, but don't expose the dashboard to the internet, and change the default passwords for anything beyond experimentation. Wazuh documents a password-change tool for exactly this.

Part 3 — Deploy an agent to a Linux endpoint

Move to the machine you want to monitor. First add Wazuh's package repository and signing key:

curl -s https://packages.wazuh.com/key/GPG-KEY-WAZUH \
  | sudo gpg --no-default-keyring --keyring gnupg-ring:/usr/share/keyrings/wazuh.gpg --import
sudo chmod 644 /usr/share/keyrings/wazuh.gpg

echo "deb [signed-by=/usr/share/keyrings/wazuh.gpg] https://packages.wazuh.com/4.x/apt/ stable main" \
  | sudo tee /etc/apt/sources.list.d/wazuh.list

sudo apt update

Now install the agent, telling it where the manager lives. The WAZUH_MANAGER variable handles enrollment automatically — point it at your server's IP:

sudo WAZUH_MANAGER="10.0.0.5" apt install -y wazuh-agent

Start it and enable it on boot:

sudo systemctl daemon-reload
sudo systemctl enable wazuh-agent
sudo systemctl start wazuh-agent

One good-practice step Wazuh recommends: after installing, disable the Wazuh repository on the endpoint so a routine apt upgrade doesn't accidentally jump the agent ahead of your manager's version (agents should be the same version as the manager or older):

sudo sed -i "s/^deb /#deb /" /etc/apt/sources.list.d/wazuh.list

That's it for a Linux endpoint. Within a minute it should appear in your dashboard.

Part 4 — Deploy an agent to a Windows endpoint

For a Windows machine, open PowerShell as Administrator and download then install the agent MSI, again pointing it at your manager:

Invoke-WebRequest -Uri https://packages.wazuh.com/4.x/windows/wazuh-agent-4.14.4-1.msi -OutFile $env:tmp\wazuh-agent.msi

msiexec.exe /i $env:tmp\wazuh-agent.msi /q WAZUH_MANAGER="10.0.0.5"

NET START WazuhSvc

The /q runs it quietly with no wizard; the WAZUH_MANAGER value enrolls it. NET START WazuhSvc starts the service. (Check Wazuh's download page for the current MSI filename if the version above has moved on.)

macOS and other platforms follow the same pattern — install the package, set WAZUH_MANAGER, start the service. The Wazuh docs have per-OS instructions.

Part 5 — Confirm your agents are reporting

Back in the Wazuh dashboard, open the Agents view. Each machine you deployed should show up as Active, with its name, IP, and operating system. If an agent is stuck on Disconnected or Never connected, the usual culprits are:

  • a firewall between the endpoint and the manager (Wazuh agents talk to the manager on TCP 1514 by default — make sure that's open),
  • a wrong or unreachable WAZUH_MANAGER IP, or
  • the agent service not running (systemctl status wazuh-agent on Linux; the service in services.msc on Windows).

Once they're green, data starts flowing. Now the useful part: knowing what to look at.

Part 6 — What to actually look at first

Wazuh surfaces a lot. If you try to watch everything at once you'll drown. Here's a sane order of priorities for a home lab or small shop.

File Integrity Monitoring — your highest-signal starting point

Wazuh watches key directories and alerts when files are added, changed, or deleted. Out of the box it monitors sensitive locations (like /etc on Linux), and the alerts tell you what changed and when. This is where you'll catch an attacker (or a misconfiguration) modifying a config or dropping a file.

To add your own path — say a directory of important scripts — edit the manager's config at /var/ossec/etc/ossec.conf and add it inside the <syscheck> block:

<syscheck>
  <directories check_all="yes" realtime="yes">/opt/important-scripts</directories>
</syscheck>

Then restart the manager (sudo systemctl restart wazuh-manager). realtime="yes" means you get alerts the moment something changes, not just on the periodic scan. Open File Integrity Monitoring in the dashboard to see the results.

Security Configuration Assessment — free hardening advice

The SCA module scores each endpoint against hardening benchmarks and lists exactly which checks pass and fail — weak SSH settings, missing password policies, risky permissions. It's like a free audit of every machine. Start with the failed checks on your most important box and work down; each finding comes with the remediation.

Vulnerability Detection — what needs patching

Wazuh inventories installed software on each agent and cross-references it against known CVEs, so you get a per-machine list of vulnerable packages ranked by severity. For a home lab this is often a wake-up call about how much unpatched software is quietly running.

Authentication and MITRE ATT&CK

Wazuh reads the endpoint's own logs and raises alerts on things like repeated failed logins, new user creation, and privilege escalation — and it maps many of these to MITRE ATT&CK techniques, so you can see not just that something happened but what kind of attacker behavior it resembles. The MITRE ATT&CK and Security events views are where this lives.

Part 7 — A concrete first win: catch a file change

Let's prove the whole thing works end to end. On a monitored Linux endpoint, touch a file in a watched directory:

echo "test change $(date)" | sudo tee -a /etc/hosts

Within a few seconds (realtime) or at the next scan, open the dashboard's File Integrity Monitoring view for that agent. You should see an event showing /etc/hosts was modified, with a before/after of what changed. Undo your test edit afterward.

That small loop — change a file, watch the alert appear — is the entire value of endpoint monitoring in miniature. Now imagine it running on every machine you own, all the time, for changes you didn't make.

Where to go next

You now have eyes on your endpoints to go with the eyes on your network. A few sensible next steps:

  • Tune the noise. A fresh Wazuh install is chatty. Spend time in the rules to silence the alerts you don't care about, so the ones you do care about stand out. An alert nobody reads is worse than no alert.
  • Prioritize by machine. Your internet-facing box and your main workstation matter more than a test VM. Harden those first using the SCA findings.
  • Set retention. Like network logs, endpoint alerts pile up. Configure how long you keep them so you don't fill the disk.
  • Be cautious with active response. Wazuh can act automatically — blocking IPs, killing processes. It's powerful and occasionally a foot-gun (it can lock you out). Turn it on deliberately, one response at a time, once you trust your alerts.

Between this and the network guide, you've built both halves of visibility on free software and spare hardware: Zeek and Elasticsearch watching the wire, Wazuh watching the machines. That combination — network plus endpoint — is the same shape a professional security operation is built on. You just run it at your own scale.
