See What's Really on Your Network: Elasticsearch, Kibana, and Zeek from Scratch

2026-07-1820 min

Most people have no idea what their network actually talks to. Your devices are making hundreds of connections an hour — DNS lookups, TLS handshakes, software phoning home, the occasional thing you'd rather know about. You can't defend what you can't see, and for a home lab or a small shop, the tooling to see it is free and runs on hardware you probably already have lying around.

This is a full walkthrough. By the end you'll have three pieces working together:

  • Zeek — a network sensor that watches traffic and turns it into rich, structured logs (who talked to whom, what DNS names were resolved, which TLS certificates were presented, and much more). Zeek isn't an alarm system; it's a meticulous note-taker.
  • Elasticsearch — a search engine that stores those logs and lets you query millions of them in milliseconds.
  • Kibana — the web interface that turns all of that into searchable tables and dashboards.

We'll wire them together with Filebeat, a lightweight shipper that carries Zeek's logs into Elasticsearch.

There's one problem everyone hits and few guides address honestly: getting traffic to your sensor in the first place. If your switch can't do port mirroring, a lot of tutorials just leave you stuck. We'll spend real time on that — it has more good answers than you'd think.

A note on scope: everything below uses example addresses like 10.0.0.0/24 and hostnames like sensor. Swap in your own. Nothing here is specific to any one network.

What you'll need

  • A machine to be the sensor/server. A spare mini PC, an old desktop, or a VM. Elasticsearch and Kibana are the heavy parts — give the box at least 8 GB of RAM (16 GB is comfortable) and a bit of disk headroom; network logs add up. You can split Zeek onto a separate low-power box later, but one machine is fine to start.
  • Ubuntu 24.04 LTS (these steps also work on Debian; adjust the Zeek repo line for your release). A clean install.
  • A way to see the traffic you care about — the whole of Part 6. Don't skip ahead and install Zeek before reading it, or it'll happily watch an interface that sees nothing.
  • sudo access and about an hour.

A quick reality check before you start: Elasticsearch is memory-hungry and will not enjoy 2 GB of RAM. If the service keeps dying on you later, RAM is the first suspect.

Part 1 — The shape of what we're building

Traffic flows past a point on your network. Zeek watches that point and writes logs. Filebeat reads those logs and ships them to Elasticsearch. Kibana reads Elasticsearch and shows you the results.

[ your network traffic ]
          │
          ▼
   ┌──────────────┐     ┌───────────────┐     ┌──────────┐
   │     Zeek     │ ──► │  Filebeat     │ ──► │ Elastic- │ ◄── Kibana (your browser)
   │ (the sensor) │     │ (the shipper) │     │  search  │
   └──────────────┘     └───────────────┘     └──────────┘

We'll build it back-to-front: get the store (Elasticsearch) and the view (Kibana) up first so we have somewhere for data to land, then set up the sensor (Zeek), then solve traffic visibility, then connect the pipe (Filebeat).

Part 2 — Update the box

sudo apt update && sudo apt upgrade -y
sudo apt install -y curl wget gnupg apt-transport-https lsb-release

Part 3 — Install Elasticsearch

Add Elastic's package repository and its signing key:

wget -qO - https://artifacts.elastic.co/GPG-KEY-elasticsearch \
  | sudo gpg --dearmor -o /usr/share/keyrings/elasticsearch-keyring.gpg

echo "deb [signed-by=/usr/share/keyrings/elasticsearch-keyring.gpg] https://artifacts.elastic.co/packages/8.x/apt stable main" \
  | sudo tee /etc/apt/sources.list.d/elastic-8.x.list

sudo apt update
sudo apt install -y elasticsearch

Read the install output carefully. Modern Elasticsearch turns on security by default the first time it installs, and during that process it prints a block that looks like this:

--------------------------- Security autoconfiguration information ------------------------------
Authentication and authorization are enabled.
TLS for the transport and HTTP layers is enabled and configured.
The generated password for the elastic built-in superuser is : <A_LONG_RANDOM_PASSWORD>

Copy that password somewhere safe right now. It's the elastic superuser login, and because the service runs under systemd rather than in your terminal, you won't get a second chance to see it scroll by.

Missed it? No problem — you can set a new one at any time:

sudo /usr/share/elasticsearch/bin/elasticsearch-reset-password -u elastic

Now start it and enable it on boot:

sudo systemctl daemon-reload
sudo systemctl enable elasticsearch
sudo systemctl start elasticsearch

Give it a minute, then confirm it's alive. Elasticsearch speaks HTTPS with a self-signed certificate, so we point curl at the certificate authority it generated and log in as elastic:

curl --cacert /etc/elasticsearch/certs/http_ca.crt \
  -u elastic "https://localhost:9200"

Enter the password when prompted. You should get back a small blob of JSON with a cluster name and a version number. That's Elasticsearch running with encryption and authentication already on — a much better default than the wide-open installs of years past.

Part 4 — Install Kibana

Kibana comes from the same repository you already added, so it's a one-liner:

sudo apt install -y kibana
sudo systemctl daemon-reload
sudo systemctl enable kibana
sudo systemctl start kibana

Kibana listens on localhost:5601 by default. If you're setting this up on a headless box and want to reach it from your laptop, edit /etc/kibana/kibana.yml and set:

server.host: "0.0.0.0"

then restart Kibana (sudo systemctl restart kibana). A word of caution: 0.0.0.0 means "listen on every interface," so only do this on a trusted network, and don't expose port 5601 to the internet.

Now connect Kibana to Elasticsearch. On the Elasticsearch box, generate a short-lived enrollment token:

sudo /usr/share/elasticsearch/bin/elasticsearch-create-enrollment-token -s kibana

That prints a long token. Open Kibana in your browser (http://<your-server-ip>:5601), paste the token when prompted, and click Configure Elastic. Kibana will ask for a verification code, which you generate on the box with:

sudo /usr/share/kibana/bin/kibana-verification-code

Enter the six-digit code, wait for setup to finish, and log in as elastic with the password from Part 3. You now have an empty but fully working search-and-dashboard platform. Time to give it something to look at.

The token is valid for 30 minutes. If it expires before you finish, just generate another one.

Part 5 — Install Zeek

Zeek is distributed for Ubuntu and Debian through the openSUSE Build Service. Add its repository (this line is for Ubuntu 24.04 — if you're on a different release, change the xUbuntu_24.04 part to match, or check the Zeek install docs for your distro):

echo 'deb https://download.opensuse.org/repositories/security:/zeek/xUbuntu_24.04/ /' \
  | sudo tee /etc/apt/sources.list.d/security:zeek.list

curl -fsSL https://download.opensuse.org/repositories/security:zeek/xUbuntu_24.04/Release.key \
  | gpg --dearmor | sudo tee /etc/apt/trusted.gpg.d/security_zeek.gpg > /dev/null

sudo apt update
sudo apt install -y zeek-8.0

We're installing the 8.0 LTS line on purpose. Zeek ships a new feature release every few months, but the LTS lines get a year of maintenance and won't surprise you by jumping to a newer major version — exactly what you want for something that's supposed to sit quietly and run.

Zeek installs into /opt/zeek. Add its tools to your PATH so you don't have to type the full path every time:

echo 'export PATH=$PATH:/opt/zeek/bin' >> ~/.bashrc
source ~/.bashrc
zeek --version

Zeek is installed. But right now it can't see anything — which brings us to the part that actually matters.

Part 6 — Getting traffic to your sensor (the part nobody explains)

Zeek can only analyze packets it can see. On a modern switched network, that's the catch: a switch is designed to send each device only the traffic meant for it. Plug your sensor into a normal switch port and Zeek will see its own traffic and almost nothing else.

The classic answer is port mirroring (also called SPAN), where you tell a managed switch to copy all traffic from one or more ports over to the port your sensor is on. If you have a managed switch, do this — it's the clean solution. Log into the switch, create a mirror/SPAN session with your uplink (or the ports you care about) as the source and your sensor's port as the destination, and you're done.

But plenty of home labs and small offices run unmanaged switches that can't mirror at all. If that's you, here are your real options, cheapest and simplest first.

Option A — Watch at the gateway (free, and often enough)

Here's the insight most guides miss: in a typical home or small-office network, almost everything interesting leaves through one place — the router/gateway. Every DNS lookup, every connection to the internet, every cloud service passes through it. If you capture there, you see all of that "north-south" traffic without spending a cent.

If your gateway runs an open firewall OS (OPNsense, pfSense, OpenWrt, or plain Linux), you can either run Zeek directly on it or mirror its WAN/LAN interface to your sensor. On a Linux-based router you can even span an interface in software with tc:

# Illustrative: mirror everything ingress/egress on the LAN interface to a dummy
# interface the sensor reads. Adjust interface names to your gateway.
sudo tc qdisc add dev eth0 ingress
sudo tc filter add dev eth0 parent ffff: matchall action mirred egress mirror dev mon0

Trade-off: you'll see everything to and from the internet, but you generally won't see "east-west" traffic — two devices talking directly to each other on the same LAN segment. For most people, north-south is 90% of what they wanted anyway.

Option B — Buy a cheap managed switch (~$30–80)

The best bang-for-buck fix is simply to replace (or add) a small "smart managed" switch that does support port mirroring. Entry-level 5- and 8-port managed switches from the usual brands cost less than a nice dinner and do SPAN just fine. Put it inline where you want visibility, mirror the uplink to your sensor, and you have a permanent, reliable tap point. For most home labs, this is the answer I'd actually recommend.

Trade-off: a small cost, and one more device in the rack. That's it.

Option C — Use a network TAP (best fidelity, some cost)

A network TAP is a small hardware device you insert inline on a cable — say, between your router and switch — that copies every packet to a monitor port. Unlike a mirror, a TAP is passive and doesn't drop frames under load, which is why it's the professional choice.

  • A passive TAP splits each direction of the link, so you may need two NICs on your sensor (one per direction) or an aggregating TAP that combines both directions onto one monitor port.
  • Aggregating TAPs start around the low hundreds of dollars — more than a cheap switch, but rock-solid and completely invisible to the network.

Trade-off: cost, and it only sees the one link you insert it on.

Option D — Make the sensor an inline bridge (free-ish, DIY)

If your sensor has two network cards, you can put it inline — traffic goes in one NIC and out the other, and Zeek watches the bridge in the middle. It's essentially a do-it-yourself TAP:

# Create a transparent bridge from two interfaces; Zeek then listens on br0.
sudo ip link add name br0 type bridge
sudo ip link set eth0 master br0
sudo ip link set eth1 master br0
sudo ip link set eth0 up
sudo ip link set eth1 up
sudo ip link set br0 up

Place that box between two segments (for example, between your switch and your access point) and it sees everything crossing it.

Trade-off: because traffic now physically passes through your sensor, if that box goes down, so does the link. Only do this where a brief outage is acceptable, and use decent NICs.

One thing to avoid

You may run across guides suggesting ARP spoofing to trick a switch into sending you other devices' traffic. Don't. It's disruptive, fragile, degrades the network for everyone on it, and it's the kind of technique that belongs in an authorized penetration test, not a permanent monitoring setup. Every option above is more reliable and doesn't fight your own network.

Picking one

  • Just want to start today, for free? Option A (watch at the gateway).
  • Want a clean, permanent, cheap setup? Option B (a small managed switch).
  • Care about not missing a single packet? Option C (a TAP).
  • Have a two-NIC box and a spot where a brief outage is fine? Option D (inline bridge).

Whatever you choose, the outcome is the same: a network interface on your sensor that sees the traffic you care about. Note its name (ip addr will list them — something like enp3s0 or mon0). That's what we point Zeek at next.

Part 7 — Point Zeek at your traffic and turn on JSON logs

First, tell Zeek which interface is your monitoring port. Edit /opt/zeek/etc/node.cfg:

[zeek]
type=standalone
host=localhost
interface=mon0        # ← your capture interface from Part 6

Next, make Zeek write JSON logs. Zeek's default log format is fine, but JSON is what Filebeat and Elasticsearch ingest most cleanly. Add one line to /opt/zeek/share/zeek/site/local.zeek:

@load policy/tuning/json-logs.zeek

Now deploy Zeek using its control tool:

sudo /opt/zeek/bin/zeekctl deploy

zeekctl deploy checks your config, installs it, and starts Zeek. Give it a moment, then confirm logs are being written:

ls -lh /opt/zeek/logs/current/
tail /opt/zeek/logs/current/conn.log

If you see conn.log, dns.log, and friends filling up, Zeek is watching your traffic. If conn.log is empty or missing, Zeek isn't seeing packets — go back to Part 6; the interface almost certainly isn't getting mirrored traffic.

Part 8 — Ship Zeek's logs to Elasticsearch with Filebeat

Filebeat is the pipe. Install it from the same Elastic repo you already have:

sudo apt install -y filebeat

Filebeat has a built-in Zeek module that knows how to parse each of Zeek's log types. Enable it:

sudo filebeat modules enable zeek

Now tell the module where Zeek's logs live. Edit /etc/filebeat/modules.d/zeek.yml — for each log type you want (connection, DNS, HTTP, SSL, and so on), set enabled: true and point var.paths at Zeek's current log directory. For example:

- module: zeek
  connection:
    enabled: true
    var.paths: ["/opt/zeek/logs/current/conn.log"]
  dns:
    enabled: true
    var.paths: ["/opt/zeek/logs/current/dns.log"]
  ssl:
    enabled: true
    var.paths: ["/opt/zeek/logs/current/ssl.log"]
  http:
    enabled: true
    var.paths: ["/opt/zeek/logs/current/http.log"]

Then point Filebeat at Elasticsearch. Edit /etc/filebeat/filebeat.yml. Because Elasticsearch uses that self-signed certificate, Filebeat needs to trust its CA:

output.elasticsearch:
  hosts: ["https://localhost:9200"]
  username: "elastic"
  password: "<the elastic password from Part 3>"
  ssl.certificate_authorities: ["/etc/elasticsearch/certs/http_ca.crt"]

Security note: a plaintext password in a config file is fine for a lab you control, but for anything beyond that, use the Filebeat keystore (filebeat keystore add ES_PWD) and reference it as ${ES_PWD} instead. Better still, create a dedicated limited user rather than using the elastic superuser. Don't ship secrets you'd mind leaking.

Load Filebeat's index templates and — the nice part — its prebuilt Zeek dashboards into Kibana:

sudo filebeat setup -e

This step talks to both Elasticsearch and Kibana and can take a minute. When it finishes, start Filebeat:

sudo systemctl enable filebeat
sudo systemctl start filebeat

Part 9 — See it in Kibana

Back in your browser, open Kibana and go to Discover (the menu is the ☰ on the left). Choose the Filebeat data view, and you should see Zeek events streaming in — one row per connection, DNS query, TLS handshake, and so on. Try a search:

  • event.dataset : "zeek.dns" — every DNS lookup on your network. You will be surprised what your devices ask for.
  • event.dataset : "zeek.ssl" — every TLS connection and the server names inside them.
  • destination.port : 23 — anything still speaking Telnet (hopefully nothing).

Because you ran filebeat setup, you also have ready-made dashboards. Open Dashboards and look for the Zeek/Filebeat set — overviews of connections, DNS, and more, with zero configuration.

That's the whole loop: traffic → Zeek → Filebeat → Elasticsearch → your eyes.

Where to go next

You now have full network visibility for the price of some spare hardware. A few natural next steps:

  • Watch DNS first. It's the highest-signal, lowest-noise place to start. Odd domains, long random-looking subdomains, and lookups to countries you have no business talking to all show up here.
  • Add retention limits. Network logs grow fast. Set an index lifecycle policy in Kibana so old data ages out instead of filling your disk.
  • Tune what Zeek logs. You don't need every protocol. Trim local.zeek to what you'll actually look at.
  • Add the endpoint half. Network visibility tells you what crossed the wire; it doesn't tell you what happened on a machine. That's the next piece — endpoint detection with Wazuh — and it's the subject of the next post.

Seeing your own traffic for the first time is genuinely eye-opening. Most networks are noisier and chattier than their owners assume, and you can't make good security decisions in the dark. Now you're not.
